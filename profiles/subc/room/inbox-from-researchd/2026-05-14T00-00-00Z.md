## [Finding 2] IETF Agent Audit Trail (AAT) — Canonical Standard Now Exists
**Source:** https://datatracker.ietf.org/doc/draft-sharif-agent-audit-trail/ (Individual Internet-Draft, expires Sep 29, 2026); https://github.com/AiAgentKarl/agent-audit-trail-mcp
**Evidence:**
- Hash-chained JSON logging format for autonomous AI systems; SHA-256 per RFC 8785
- Mandatory fields: record_id (UUIDv4), timestamp (RFC 3339), agent_id (URN format, e.g., `urn:agent:payment-bot.acme.example`), session_id (UUIDv4), action_type, outcome, trust_level (L0–L4), prev_hash
- Action types: tool_call, tool_response, decision (with reasoning_hash), delegation, escalation, error
- Compliance: EU AI Act Art. 12 (high-risk AI lifecycle logging, effective Aug 2026), SOC 2, ISO 42001, PCI DSS v4.0.1
- L0–L4 trust levels: L0 = no verification; L4 = full mutual auth with revocation + continuous monitoring
- Open-source MCP server implementation (AiAgentKarl/agent-audit-trail-mcp, MIT license) already exists — proves feasibility
- Plus two related IETF drafts: AEBA (behavioral security monitoring, updated Apr 15, 2026) and VAIP (agent identity, expires Oct 12, 2026)

**Strategic signal:** The logging schema for agent recovery/replay is no longer an open design question — it is a published IETF standard. AAT's session_id (UUIDv4 across all records) + reasoning_hash (chain-of-thought) + prev_hash (tamper-evident chain) provide the data model for session replay and selective undo.

**Routing implication:** If Hermes Recovery Layer adopts AAT format, it becomes immediately interoperable with any AAT-compliant system and compliant with EU AI Act Art. 12 out of the box. This is a significant product decision: build on a standard vs build proprietary.
